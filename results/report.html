Calendar-preconfigured Collective Communication

详细数据流动示意图见 dataflow.html（各 collective 最低 makespan calendar 及故障重调度）。

Problem setup

12×16 mesh2d (192 nodes). Horizontal link latency 4 cycles, vertical 8 cycles. PE↔router ramps: 1 flit/cycle, 1-cycle latency. NoC: 1 flit/cycle per link. In-network combine (reduce) and fork (broadcast) enabled. Calendar = contention-free TDM preconfiguration.

Gather 无阻塞下界推导

Period（稳态 initiation interval）

period = (N−1)×M = 191 — root 侧带宽瓶颈，与路径长度无关。

Makespan 下界（两种表述）

  1. 精确 slot 分配式（仿真采用）：将 191 个 flit 按路径延迟 Lᵢ 升序排列， 为第 i 个分配 root 完成时刻 Fᵢ = t₀+i，约束 Fᵢ ≥ Lᵢ。 最小化 max Fᵢ 得 t₀ = maxᵢ(Lᵢ−i) = 15，故 makespan = t₀ + (N−2) = 205。 全局 link-time calendar 反向预约各 hop 的 send-time，可达该下界（eff=1.0）。
  2. 闭式近似（旧文档 204）： (N−2) + max_path − mesh_diam + H + V = 190 + 166 − 164 + 12 = 204。 其中 max_path=166（最远源 up+mesh+down），mesh_diam=164。 +H+V 项假设 mesh 链路可额外流水线重叠 1 cycle；在本拓扑精确 slot 式下界为 205，闭式少 1 cycle。

AllGather 理论再分析：为何不是 gather + broadcast

AllGather 要求每个节点最终持有全部 N 份数据。把它当作 「先 gather 到 root，再从 root broadcast」会得到 205 + 166 = 371， 但这高估了下界——两阶段被人为串行化，且所有数据被迫汇经 root 形成热点。

正确下界 = 三个约束取最大

  1. down-ramp 带宽下界：每个节点的单条 down-ramp 必须吞入 其余 (N−1) 个源的数据 = (N−1)×M = 191×M。这是每个节点都同时 承受的瓶颈（gather 中只有 root 承受）。
  2. bisection 下界：最小割 12 条链路，半数节点数据需跨割 = ⌈(N/2)×M / 12⌉ = 8×M（本拓扑非瓶颈）。
  3. 最坏接收者 pipeline：最远角节点接收 (N−1)M flit 的 gather slot 下界 = 205(M=1)。

三者取大：max(191, 8, 205) = 205(M=1)。 关键结论：allgather 下界 ≈ gather 下界（205），而非 gather+broadcast（371）。 直觉上 allgather = 「N 个并发 gather」，其代价由最坏的那一个 gather 主导， 额外的副本是并发送达的，而不是在一次完整 gather 之后再串行广播。

实现的最优算法：双向维序多树（bidirectional dimensional multi-tree，命中下界）

把行/列遍历重叠到单阶段：每个源沿 X-先-Y 维序展开一棵组播树 ——行脊（H-link 双向）+ 每列分叉（V-link 双向）。注意没有字面意义的 Hamiltonian 环， 每个维度上的「双向 line」就是 mesh 上环的等价物。转发为 router 内 fork： 节点把到达 flit 复制一份下 down-ramp（eject 到 PE），另一份继续转发， 中间节点从不 eject 后再 reinject，因此不付 10cy 的 PE/SRAM bounce。

N 棵树叠加后用全局 link-time calendar 贪心装填：每条有向 link、每个 down-ramp 每周期 ≤1 flit，构造即无冲突。最坏角节点 (0,0) 的列0 漏斗承载 180M flit、 down-ramp 吞 191M flit，正是瓶颈。仿真（C++ ScheduleAllGatherDimMultiTree + utils/sim_dim_multitree.py）确认 makespan 精确命中下界： 205 / 769 / 3061 / 12229（M=1/4/16/64），eff=1.0。

对比：2D dimensional allgather（次优，两阶段）

另一可行算法是先沿行（Phase X，H-link）做行内 allgather（makespan 46）， 再沿列（Phase Y，V-link）做列内 allgather（makespan 189），down-ramp 之和同为 (N−1)M。 但两阶段被串行化，总 makespan = 46+189 = 235(M=1)，比下界高 30 cycle （Phase X 的 fill 未被隐藏）。多树方案把这段 fill 重叠掉，故能从 235 收紧到 205。

AllGather：无冲突、缓冲与 makespan 权衡（真实物理约束）

「无冲突」≠「无缓冲」

Calendar 保证的是同一 cycle、同一资源上最多 1 个 flit： 每条有向 link 每 cycle 最多 1 次注入（inject≤1）， 每个 router 的 down-ramp 每 cycle 最多 1 次 eject（eject≤1）。 这与「flit 从不排队」是两回事——若不加约束，争用会被暂存进缓冲（输出端口 FIFO 或 down-ramp 入口队列）后再错开放出。

两条硬性物理约束

真实硬件上这两类缓冲都不能随意存在：

  1. down-ramp / eject 缓冲必须为 0：flit 到达目的地当 cycle 必须被 PE 取走， 不允许在 down-ramp 入口排队。形式上 eject_cycle == arrive_cycle，down-ramp 仅在该 cycle 占用。
  2. 每跳网内等待必须有界（W 有限）：flit 在输出端口最多滞留 W cycle。 无上限滞留会与后续 flit 踩踏——被卡住的 flit 占住端口/链路，后面 wormhole 流水的 flit 会撞上它。

注意：205 本身仍是最优 makespan（见下「真实下界」），它没有问题；有问题的是实现它的那个贪心日历 ——它靠角节点 50–87 flit 的 eject 队列 + 输出端口无界滞留来达到 205，而这正是上面禁止的踩踏场景。 所以问题不是「205 不可达」，而是「在 E=0 且 W 有界下，我们目前的调度器还达不到 205」。下面给出满足约束后的真实结果。

三类「同时到达」场景（区分冲突与合法并发）

  1. 左右邻居同时到达 router（不同输入端口）——合法，arrive 可 >1，不是冲突。
  2. 两者都要 fork 到同一输出端口——真争用；靠错开 send 周期化解（该 lane 全程 inject≤1）， 而非靠无界排队。
  3. 同一 link 上多个 flit 同时在飞（inFlight>1）——wormhole 流水线，非阻塞； 各 flit 在不同 cycle 注入、沿 link 错开飞行。

真实下界：release-time packing（关键修正）

每个节点单条 down-ramp 必须吞 (N−1)M flit、每 cycle 1 个，给出带宽项 (N−1)M+ramp = 48(6×8)/192(12×16)。但这不可达：down-ramp 不能在 flit 物理到达前 eject， 而到达时刻被 mesh 距离量化——源 s 到节点 d 最早到达 ramp + dist(s,d)（injection 只能推迟、不能提前）。 于是每个节点的 down-ramp 是一个带 release-time 的单机排程（N−1 个单位作业按 release 升序贪心装填）， 其 makespan 取所有节点的最大 = release-packing 下界 LB*。

实测（utils/sched_zero_eject_v2.py 的 packing_lb）： LB* = 78（6×8）/ 205（12×16），瓶颈都在最远角节点 (0,0)。 注意 LB* 是松弛下界——它只看单个 down-ramp 的 release-packing，忽略了链路争用与 fork 树的共享前缀耦合。 真正的最优（同时满足 E=0、链路无冲突、有界等待）由下面的 CP-SAT 精确求解给出，会略高于 LB*。

用 CP-SAT 精确求解 buffer-free / conflict-free / non-blocking 排程

把问题写成约束规划（utils/sched_ilp.py，OR-Tools CP-SAT）：变量为每个 (源, 节点) 的到达 cycle a[s,d]；a[s,s]=inject+ramp；树边 a[s,p]+lat ≤ a[s,c] ≤ a[s,p]+lat+W（因果 + 有界每跳等待＝non-blocking）； 每条有向链路 AllDifferent（发送 cycle 互异＝无冲突）；每个节点 down-ramp AllDifferent（eject 互异、E=0、无队列）； 目标 min makespan。结果：

满足约束的排程（utils/sched_no_eject_buffer.py）

调度器强制 eject 缓冲 E=0（到达即取走），并用每跳 ≤W 的网内等待来错开最后一跳的到达 cycle， 使下行端口无需排队；若 W 内找不到可行时隙，则把整条源树的 injection 整体后移 （争用退回源 PE 发射队列，这不属于网内/eject 缓冲）。全程 inject≤1、eject≤1、E=0、每跳等待 ≤W。

down-ramp 带宽扩充（B flit/cycle eject）

若把每个节点的 down-ramp 从 1 flit/cycle 扩到 2 或 4 flit/cycle（仍 E=0：到达当 cycle 即 eject，只是同一 cycle 可并行取走 B 个 flit）， 链路带宽仍为 1 flit/cycle/directed-link。Makespan 由两个下界取大决定：

makespan ≥ max( ⌈(N−1)M / B⌉ + ramp , latency_floor )

  • eject 带宽项 ⌈(N−1)/B⌉+ramp：随 B 增大而下降（6×8：48→25→13；12×16：192→97→49）。
  • 延迟地板 latency_floor：最远 flit 的物理传播 + ramp，与 B 无关 （6×8 = 78；12×16 = 166，角 (11,15)→(0,0) 的维序路径 11×H+15×V+2×ramp）。

结论（CP-SAT 精确求解后的最终结论）

  • 零 eject 缓冲 + 无冲突 + non-blocking 是可行的，且 CP-SAT 能构造出来并证明（近）最优： 6×8 真实最优 = 81（已证），12×16(W=8) ∈ [257, 313]。
  • 它不是免费的（修正早前基于松弛 LB* 的乐观结论）：相比带缓冲乐观值， 6×8 多 +3（78→81，约 4%），12×16 至少多 +52（205→≥257，约 ≥25%，W=8）。 代价来自 fork 树共享前缀耦合 + 链路争用，松弛 LB*（只看单 down-ramp）看不到这部分。
  • CP-SAT 明显优于贪心：6×8 把 W=4 的 108→81、W=8 的 103→81（已证最优，约 −25%）； 12×16(W=8) 把贪心 330 改进到 313，并把下界收紧到 257。
  • W=4 对 6×8 已足够命中最优 81（W=0 受限到 ~118、W=2 到 ~86）；非阻塞的有界等待不需要很大。
  • 规模瓶颈：12×16 是 ~37k 变量 + 192 组大 AllDifferent，CP-SAT 600s 内 从零找不到可行解；必须用贪心解热启动（--warmstart，AddHint + 收紧 horizon）才能在 313 上继续改进。 要逼近真实最优需更长时间或问题分解（按瓶颈节点分块 / 列生成）。
  • down-ramp 加宽有效但很快饱和：B=1→2 在 12×16 上 LB* 205→166、贪心 274→201（−27%）； 6×8 上 CP-SAT 81→78。B≥2（6×8）/ B≥2（12×16）均已撞上延迟地板（78/166），继续加宽无收益。 脚本 utils/sched_eject_bw.py；CP-SAT 加宽用 sched_ilp.py --bw B（cumulative 约束）。
  • 相关脚本：utils/sched_ilp.py（CP-SAT 精确/热启动，需 ortools，见 .venv-ilp）、 utils/sched_eject_bw.py（down-ramp 带宽 B 扫参）、 utils/sched_zero_eject_v2.py（松弛下界 LB*）、 utils/sched_no_eject_buffer.py（贪心 E=0 + 有界 W：274/330）、 utils/sched_zero_buffer.py（W=0 完全刚性 139/579）。

Q1: Minimum makespan and calendar period

Q1 conclusion: Not all collectives can reach the unconstrained latency minimum.

  • broadcast: root up-ramp inject + mesh tree fork + leaf down-ramp; optimal period M.
  • reduce: leaf PE nodes inject via up-ramp; inline combine on mesh; root down-ramp for final result; period M.
  • allreduce: reduce phase (up+mesh+down) then broadcast phase (up+mesh+down).
  • gather: period 191×M (root down-ramp); makespan bound maxᵢ(Lᵢ−i)+(N−2) with global link-time calendar (M=1 healthy: 205).
  • allgather: the optimum is NOT gather+broadcast. True bound = max((N−1)×M down-ramp, bisection, worst-corner gather) = 205 (M=1), not 371. The bottleneck is every node's single down-ramp absorbing (N−1)×M flits — allgather costs essentially one worst-case gather, not a gather plus a broadcast. The implemented optimum is the bidirectional dimensional multi-tree (X-then-Y in-network fork trees superposed under one global link-time calendar; no literal ring): it hits the lower bound exactly — makespan 205 (M=1) → 12229 (M=64), efficiency = 1.0, conflict-free.
  • alltoall: bisection bandwidth bound 764×M plus diameter path drain (+164+2 ramp cycles); anytoany uses full per-hop calendar simulation.
