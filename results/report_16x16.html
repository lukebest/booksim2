16×16 Mesh Allgather 方案对比（H=4, V=6）

问题设定

16×16 mesh（256 节点）。横向 link delay H=4 cycle，纵向 V=6 cycle，PE↔router ramp 延迟 1 cycle。下 Ramp（eject）带宽分别取 1 与 2 flit/cycle 两种场景，msg_size=1。

两套调度模型（报告中始终分开比较）：

  • 严格 0-buffer 刚性：所有时隙离线编排，无冲突（每条有向 link 每 cycle ≤1 flit、每节点下/上 ramp ≤ramp 带宽）、无阻塞（全离线）、路由器零缓存——flit 一旦注入即按固定时刻逐跳前进，中间绝不等待；唯一自由度是每源注入偏移（数据暂存在源 PE/SRAM，非路由器 buffer），贪心打包出可行上界。脚本 utils/sched_zerobuf_compare.py。
  • 时分 / 流水：事件驱动全局 link-time calendar，允许同一条 link 被不同源 flit 时分复用、并允许网内缓存——这是流水的理想时延（下界侧）。脚本 utils/sim_fused_rings.py。

通用下界：每节点都要经其单条下 ramp 下泄 N−1=255 条 flit，故任何方案 makespan ≥ (N−1)/ramp_bw + 最小投递延迟（BW=1→255，BW=2→128）。

五类方案

  • dimensional multi-tree：每源用 X-then-Y 维序多播树（行脊 + 各列分支，网内 fork），带 buffer 时可命中下界。
  • 纯 Hamilton 环：全局一个蛇形 comb 闭环，单向 / 双向。
  • hybrid 局部环 + 全局树：按行切 B 个水平带（每带 R=MY/B 行）；①带内跑局部 Hamilton 环 allgather（并行）；②每列向上下相邻带做树状广播互换各带块。
  • quad 4×(8×8) 环 + 中心交换：切成 4 个 8×8 象限各跑环；4 象限最内角在中心构成 4-环，象限块经中心时分互传，对端再绕环二次分发。
  • border 4×(8×8) 环 + 边界多点注入：同样切 4 象限跑环，但外部数据沿象限共享边界的多点跨界，对端只需沿行/列短弧分发（≪一圈），逼近“一圈 + 尾”。

汇总：各方案 makespan（按下 Ramp 带宽 × 调度模型）

两套模型分开看：0-buf 刚性＝严格 0-buffer、刚性时隙（路由器零缓存，仅源端可缓存）；时分＝允许网内时分复用 + 源端缓存的流水/理想时延（同一引擎计算，可直接比较）。同一方案两列之差 = 0-buffer 的代价。绿底＝该列最小（越低越好）。

数据洞察

  1. 0-buffer 的代价 = 足迹局部性。同一方案“0-buf÷时分”的倍率：multi-tree 837/265≈3.2×（足迹横跨整网，刚性最难错开）；hybrid 行带 ≈1.6×（足迹最紧凑，最抗 0-buffer）；quad 单向 717/715≈1.0×（本就延迟受限，刚性化几乎不再变差）。局部性越强 → 越抗 0-buffer。
  2. 时分模型里最优是 multi-tree 与 hybrid，不是 border。时分 BW=1：multi-tree 265、hybrid 265 已贴 eject 下界 255；border 双向 283 紧随其后，是“环局部性”系列里的最优，但仍略逊于树/行带。（此前把时分 border 与 0-buf 方案混比的说法已纠正。）
  3. 下 Ramp 带宽 1→2 是否有用，取决于瓶颈。受“下泄”约束者才受益：0-buf multi-tree 837→504、0-buf quad 双向 1097→523、时分 border 双向 283→267；受“延迟/链路”约束者几乎不变：纯环恒 1474/754、quad 中心恒 715/717、时分 multi-tree 265→255（已链路受限）。一旦调度逼近延迟最优，2 flit/cy 下泄几乎白费。
  4. 方向（单/双）是双刃剑。时分里双向普遍把最坏路径减半（纯环 1474→754、border 437→283、quad 中心 715→413）；但 0-buf 刚性里双向中心交换有害——quad 单向 717 vs 双向 1097（BW=1），双向制造更多链路冲突反而更慢。
  5. “一圈 + 尾”得到验证。8×8 独立环单向 354、双向 186；border 16×16 单向 437≈354+尾、双向 283≈186+尾；中心单点 715≈2×354 是两圈。每条环 link 仅需承载 255 flit < 一圈容量 356，带宽够、是延迟（圈数）决定胜负。

结论（已复核）

  • 允许 buffer / 时分时：multi-tree 与 hybrid 行带并列最优，均贴近 eject 下界（BW=1 约 265/265，BW=2 约 255/255）；border 边界多点（283/267，双向）是象限局部方案中的最优、紧随其后；中心单点（413/715）与纯环（754/1474）最差。
  • 严格 0-buffer 刚性时：hybrid 行带整体最优。BW=1 ≈ 416(B=2)，比 multi-tree 的 837 快 2.01×；BW=2 ≈ 409(单向 B=8)，优于 multi-tree 504。原因：行带足迹时间跨度小、占用稀疏，最易刚性错开。
  • multi-tree 在 0-buffer 下退化（837/504）：每源足迹横跨整网（直径 150 cycle），又宽又满，256 个宽足迹难以单偏移错开 → 约 3× 下界。
  • quad 中心交换居中：单向恒 717（延迟受象限环周长主导，加带宽无效），双向 0-buf 反而更差（BW=1 1097）。可行、规整、对称，但中心 4-环是唯一跨象限通道，受两段绕行延迟约束。
  • 选型：可缓存 → multi-tree / hybrid；硬性 0-buffer 刚性 → hybrid 行带；要规整 4 象限版图又接近最优延迟 → border 边界多点（时分）。

时分融合分析：8×8 环单元 + 4 环中心交换的“一圈 / 两圈”问题

模型：事件驱动的全局 link-time calendar（无冲突、网内 fork、每条 link 1 flit/cy、每节点下 ramp ramp_bw flit/cy），允许同一条 link 被不同源的 flit 时分复用（即“时分地插入其他环的数据”）。这是流水/时分的理想时延；严格 0-buffer 刚性版即上文的 quad（717/1097）。脚本 utils/sim_fused_rings.py。

关键结论：中心单点 = 两圈；边界多点 = 一圈 + 尾

  • 带宽上一圈本就够：每条环 link 仅需承载 局部 63 + 外部 192 = 255 flit < 一圈容量 356，带宽不是瓶颈。
  • 中心单点注入 → 延迟必须两圈：外部数据只能从中心 4 角进入，进对端环后要覆盖其 64 个节点必须再绕近一圈，最坏路径 ≈ 自身环 1 圈 + 对端环 1 圈 = 2 圈（实测单向 715≈2×354）。
  • 边界多点注入 → 真正做到“一圈 + 尾”：外部数据沿共享边界 8 点同时跨界，对端只需沿行/列短弧（≤7 跳）分发。最坏路径 ≈ 自身环 1 圈 + 跨界 + 短弧 ≈ 一圈 + 小尾。实测单向 437（≈8×8 一圈 354 + 83 尾），较中心方案 715 提速 1.64×；busiest link 从 255 降到 191（负载也更均衡）。
  • 边界 + 双向是象限局部方案里的最优：BW=1 283、BW=2 267，远优于同模型下的中心单点（413/715）与纯环（754/1474）；不过在时分模型里它仍略逊于 multi-tree 与行带 hybrid（≈255–265，见上方汇总表）——后两者已贴 eject 下界。（注意：不要把时分 border 与严格 0-buffer 方案直接比，两者模型不同。）
  • 代价：边界多点注入用满了象限共享边界的 8 条跨界链路并在对端做行/列分发——本质上是把“环 + 局部树”结合，结构比“中心 4 环”略复杂，但仍是规整的象限化布局。
