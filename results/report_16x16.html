16×16 Mesh Allgather：0-buffer 方案对比（H=4, V=6）

问题设定

16×16 mesh（256 节点）。横向 link delay H=4 cycle，纵向 V=6 cycle，PE↔router ramp 延迟 1 cycle，msg_size=1。下 Ramp（eject）带宽取 1 与 2 flit/cycle 两种场景。

硬约束：所有方案必须严格 0-buffer——所有时隙离线编排，无冲突（每条有向 link 每 cycle ≤1 flit、每节点上/下 ramp 每 cycle ≤ ramp 带宽）、无阻塞（全离线、无运行期阻塞）、路由器零缓存（flit 一旦注入即按固定时刻逐跳前进，中间节点绝不等待；多播扇出按“到达即组合复制/直通”，扇出计入单跳延迟）。唯一自由度是每源的注入偏移（数据暂存于源 PE/SRAM，非路由器 buffer）。需缓存才能更快的“时分/流水”方案已全部移除，本报告只比较 0-buffer makespan。脚本 utils/sched_zerobuf_compare.py，数据 results/zerobuf_16x16.json。

通用下界：每节点都要经其单条下 ramp 下泄 N−1=255 条 flit，故 makespan ≥ (N−1)/ramp_bw + 最小投递延迟（BW=1→255，BW=2→128）。

参与比较的方案（均 0-buffer）

  • dimensional multi-tree：每源 X-then-Y 维序多播树（行脊 + 各列分支，网内 fork）。
  • 纯 Hamilton 环：全局一个蛇形 comb 闭环，单向 / 双向。
  • hybrid 横带环 + 纵向树：切 B 个水平带（每带 R=MY/B 行），带内跑横向 Hamilton 环 allgather，再每列向上下相邻带做纵向树广播。
  • hybrid 纵带环 + 横向树（新增）：转置版——切 B 个垂直带（每带 C=MX/B 列），带内跑纵向 Hamilton 环 allgather，再每行向左右相邻带做横向树广播。
  • quad 4×(8×8) 环 + 中心交换：4 象限各跑环，最内 4 角构成中心 4-环互传，对端再绕环二次分发。
  • border 4×(8×8) 环 + 边界多点注入：4 象限各跑环，下方两环上下翻转使每个环的长边(梳脊)都贴中心边界，横向边界变为脊对脊、跨界注入点更多；外部数据沿共享边界多点跨界 + 短弧分发（亦为多播）。

border 时分注入泳道图（多 Hamilton 环，下 ramp=1）

单向绕环时，相邻 flit 沿环按一跳到达（均跳延迟 ≈ (H+V)/2 = 5 cycle），故某 router 的下 ramp 每 5 拍只有 1 拍下泄本环数据，余下 4 拍空闲。其它 3 个环把数据算好提前量、在边界（脊对脊）跨界注入到这些空槽里——离得近的邻环先到、对角环最后到。稳态下每个 5 拍窗口内集齐 4 种颜色，即每个环都拿到全部 4 个环的数据。

每格＝该 router 在该 cycle 下泄的 flit 所属环（数字/颜色＝源环号）；灰“空”格＝该拍下 ramp 空闲、可被其它环的提前量注入占用。本图为时序示意，用于解释“长边贴边界”后跨界注入如何填满空槽；严格 0-buffer makespan 见汇总表。

数据洞察

  1. 朝向决定胜负（H=4 < V=6）：纵带环 + 横向树最优。BW=1 它做到 334，优于横带环+纵树 416、multi-tree 837；BW=2 同样以 334 领先（409/504）。原因：把“最长的那段树”放到便宜的水平方向——纵带版树跨度最多 12×H=48，横带版树跨度 12×V=72，刚性错开更省。环脊则相反（纵环脊走贵的 V），但综合下来横向树的节省占优。
  2. 带数 B 很敏感。纵带环+横向树最优在 B≈2~4（C=8~4），横带环+纵向树最优在 B≈2（BW1）/B=8（BW2）；选错 B 可差 2~3 倍（见明细表）。把局部环做小、树做短，整体更易刚性错开。
  3. multi-tree 在 0-buffer 下退化：837/504（≈3.3× 下界）。每源足迹横跨整网（直径 150 cycle），又宽又满，256 个宽足迹难以单偏移彼此错开。
  4. 方向是双刃剑。横带 hybrid 双向在 BW1 有利；但 quad 双向 0-buffer 反而更差（单向 717 vs 双向 1097，BW1）——双向中心交换制造更多链路冲突。
  5. 带宽 1→2 只帮“受下泄约束”的方案：multi-tree 837→504、quad 双向 1097→523 明显下降；而纯环（1474/754）、quad 单向（恒 717）受延迟约束，加带宽无效。最优的纵带 hybrid 两带宽同为最优，说明它已不受下泄瓶颈限制。

结论（仅 0-buffer）

  • 0-buffer 最优方案：hybrid 纵带环 + 横向树。BW=1 / BW=2 均为 334 / 334，优于横带环+纵向树（416/409）、multi-tree（837/504）、quad（717/717）、border（437/437）、纯环（1474/754）。
  • border “长边贴边界” 大幅提速：把下方两环上下翻转、令 4 条梳脊都贴中心边界后，横向边界变为脊对脊、跨界注入点增多、时序更整齐——border 单向 BW=1 从 782 降到 437（约 1.8× 提速），已接近行带 hybrid。
  • 关键洞察：在 H<V 的网格上，应让“树/多播”走便宜的方向。本例 H=4<V=6，故纵向 Hamilton 环 + 横向多播树优于横向环 + 纵向树。若 H>V 则结论相反。
  • multi-tree 虽在可缓存模型最强，但严格 0-buffer 下退化到 837/504（足迹太宽，难刚性错开）。
  • quad / border 结构规整、4 象限对称：quad 中心交换偏慢（单向恒 717，中心通道单一）；border 长边贴边界后已显著改善（单向 437/437），是规整象限版图里的最优折中。
  • 选型（硬性 0-buffer）：首选 hybrid 纵带环 + 横向树；要规整 4 象限版图 → border 长边贴边界。

边界 link 为 AFIFO：包经过边界节点即可多播写入，不必等本环整圈结束。四象限环并行运转，均衡发送即可。脚本 utils/sim_afifo_border.py。

为什么当前「全圈 foreign lap」仿真 ≈ 3×T_lap？

当前 build_afifo_delivery 在边界分叉（仿真已正确：到达边界同一 cycle 继续本环 + 写入 AFIFO），但进入邻环后仍要求整圈遍历 H / V / 对角环。对最坏源 s，对角链为：

t ≈ d_H + H + d_Vh + V + T_lap(diag) + RAMP

其中 d_H、d_Vh 可达 O(T_lap)，再加对角整圈 T_lap → L_sync ≈ 3.0×T_lap（16×16 单向 1048/352）。且 256 源同时在 t=0 注入，link 日历争用 → AFIFO sim = 1055 cycle。

这不是「必须等本环跑完才能跨界」——分叉已在边界即时发生；3× 来自邻环整圈叠加最坏路径，而非本环串行等待。

均衡 AFIFO 下的 makespan 下界：T_lap + τ（一圈 + 尾）

正确思路：四环同时跑各自的一圈；本环包未到边界前的空闲时隙由 AFIFO 读入邻环数据；跨界后在邻环只需短弧覆盖（沿行/列从边界扩散），不必再绕完整 foreign 整圈。均衡发送 = 各源注入/跨界时刻分散在 [0, T_lap] 窗口，避免挤在同一边界 AFIFO。

T_min = max( T_lap + τ, B_eject )

  • T_lap：单包单环整圈链路时间（四环并行，取 max 亦为 T_lap）
  • τ：最后一批边界 AFIFO 注入后，沿邻环短弧抵达最远节点所需尾延迟。16×16 单向实测 τ = border_ref − ring_ref = 83 cycle
  • B_eject = ⌈(N−1)/r⌉

16×16 单向：T_min = 352 + 83 = 435 （border 短弧参考 437 cycle），远低于全圈 foreign lap 仿真的 1055 cycle。

16×16 结论

  • 3×T_lap 的原因：foreign 整圈模型在最坏源上对角链 ≈ 3 个环长；不是「等本环完成才跨界」。
  • 最小 makespan（均衡 AFIFO）：T_lap + τ ≈ 352 + 83 = 435 cycle（单向），四环并行一圈，边界 AFIFO 填空闲时隙，邻环短弧收尾。
  • 全圈 foreign 仿真 1055 是未做均衡短弧优化的上界，不应与 T_min 混淆。
  • 实现 T_min 需：边界即时分叉（已有）+ 邻环短弧而非整圈 + 注入时刻均衡分散（见 balanced_inj_offsets）。
