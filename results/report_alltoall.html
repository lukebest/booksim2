All-to-All / Any-to-Any：零 router buffer 最优模式 & buffer–makespan 关系

基于 booksim2 calendar 模型（12×16 mesh, H=4, V=8, ramp=1, 每有向 link 1 flit/cycle）。 相关仿真见 report.html、数据流见 dataflow.html。 生成脚本 utils/gen_alltoall_report.py。

1. 约束定义

无冲突每条有向 link 每个 send-cycle 最多 1 flit（inject≤1）。

无阻塞每跳在 router 输出端口最多等待 W cycle；W=0 为完全刚性。

无 router bufferring_buf=0 且 eject_buf=0：flit 不在 router 排队； 争用只能退到源 PE 注入队列（或边界 AFIFO，mesh all-to-all 通常无 AFIFO）。

E=0到达目的节点当 cycle 必须 eject，down-ramp 无队列。

2. All-to-All 最优算法与 Pattern

理论下界

瓶颈在二分带宽，不是直径：

makespan ≥ ⌈N(N−1)M / B⌉ + diameter + 2×ramp − M

12×16（N=192, B≈48）：带宽项 764×M；pipeline drain ≈ 166−M → M=1 完整下界 929 cy，M=16 带宽项即 12224 cy。

最优路由 Pattern

维序最短路径（XY 或 YX）：先走完 Δx（H-link），再在转角 router 切到 Δy（V-link）。 固定、无死锁、无 fork——与 AllGather 的多播树是不同问题。

最优调度

  1. 二分槽流水线（collective.cpp::PlanAllToAll）：对 N(N−1)M 个流编号， 按 start = slot / B 错开注入，使最小割平均 B flit/cycle。
  2. 两阶段维序（文档规划）：Phase-X 完成全部水平位移 → Phase-Y 完成垂直位移； 大 M 时与槽流水线等价，均逼近 764×M。

零 buffer 要点

每条 (src,dst) 路径在给定 inject_s 后是刚性链： send(p→c) = arrive(p)（W=0）。全局只需为所有流选注入相位，使 link send-cycle 不碰撞—— 本质是边着色 / link slot packing，比 AllGather 的多播树耦合简单得多。

5. Router buffer 深度 vs All-to-All makespan

5.1 核心结论：All-to-All 对 buffer 不敏感

  • Calendar 调度器冲突时 ReserveLink 会 t++ 等待 → 隐式 W=∞。
  • 大 M 时 makespan 已精确命中二分带宽下界；加深 buffer 无法进一步降低。
  • Unlike AllGather：无 fork 共享前缀、无每节点 eject (N−1) 份多播副本的强耦合。
  • Router buffer 主要用于 XY 转角 1–2 cycle 对齐；W≥2 预期即饱和。

5.5 设计建议

  • 大 M：优化二分槽流水线 + XY 路由，不必为 all-to-all 堆深 router FIFO。
  • 小 M：report 中 eff<1 是相对纯带宽项 764×M 的比值；含 drain 的完整 makespan 在 M=1,4 已达标。
  • 强制 W=0：预期小幅 overhead 后仍贴近 764×M + drain 平台。
  • 缺口：本仓库尚无 all-to-all 专用 W 扫描脚本（可参考 sched_buffer_sweep.py 改为单播路径）。

6. 总结

  • All-to-All 零 buffer 最优：XY 单播 + 二分槽注入 + 全局 link calendar；勿用 AllGather 多播树或 Hamilton 环。
  • Any-to-Any：预计算置换 + 最短路径；小 M 按路径贪心，大 M 同 all-to-all 槽流水线。
  • Buffer–makespan：曲线早饱和、惩罚远小于 AllGather；M 大时 makespan 由二分带宽锁定，与 W 无关。
