  2. 严格 0 router 缓冲（K=0）：单环最优（ramp_bw=2/4 → 754 bi；ramp_bw=1 → 1474 uni）。 K≤2 时 ramp_bw=2 → border 267；ramp_bw=1 → hybrid B=2 418（border 需 K≤74 才到 283）。
  3. 允许大缓冲：grid 8×2 在 ramp_bw=2、K≥56 时 mk=189，逼近 fill；eject 地板 128 仍不可破。
  4. 最优切分层数 Q* ≈ H = 4：再细切（8×2）需 K≈30~56 才值得。

1. 形式化模型

  • 每条有向 mesh 链路：≤1 flit/cycle 发射槽，latency H 或 V（流水线，可同时 in-flight L 个 flit）。
  • 每节点下 ramp：≤ ramp_bw flit/cycle 到达/eject；上 ramp 注入源 flit。
  • Router 内 in-network fork：共享输入链路只计 1 次载荷；下 ramp 不做多播复制。
  • Router 缓冲预算 K：max(link_buf, ramp_buf) ≤ K；边界 AFIFO 单独计数（本报告 Pareto 只约束 router 缓冲）。
  • pipe_mk：允许足够缓冲时的冲突无关流水线 makespan（sim_fused_rings.simulate）； peak buffer 由 measure_buffers 统计。
Teject = ⌈(N−1) / ramp_bw⌉   |   Lmax = max有向边 e |{源 s : s 的树路径经过 e}|   |   fill = maxs,v latency(inject→v) + 2·ramp

无冲突 makespan 下界：max(fill, Lmax, Teject)。实测 pipe_mk 紧贴该 max（见明细表「主导」列）。

1b. 术语说明：B、ramp_buf 与下 ramp 突发

B（hybrid B=k 中的带数）

B 是把 mesh 在纵向（行方向）均分成 B 个水平带的个数。 对 16×16 mesh，行数 MY=16，每带高度 R = MY/B 行：

B带数每带规模本带 Hamilton 环跨带传播
22 条水平带16×8源在本带内跑 16×8 蛇形环（①）每列再向上下相邻带做纵向树（②）
44 条16×4同上，环更小列树跨度更短
88 条16×2环更小、fill 更低树更浅，但 eject 突发更大
1616 条16×1退化为单行环几乎只剩列树 → 接近 multitree

代码：build_hybrid_delivery(s, B, bidir) — R = MY // B，ham_cycle_rect(0, y0, MX, R) 为本带环，随后对每列 x 向带外行做链式 fork（纵向树）。 B 越大 → 局部环越短（fill↓），但根附近分叉越早、越多源 flit 同时涌向同一节点的下 ramp（ramp_buf↑）。

ramp_buf（下 ramp 侧 router 缓冲深度）

ramp_buf 是在冲突无关流水线日历下，某一个节点在其下 ramp 口上，所需 eject 缓冲的峰值深度（单位：flit）。统计方式（measure_buffers）：

  1. flit 经 mesh 到达目的节点 c 的时刻为 arrive；
  2. 下 ramp 日历把该 flit 排到可 eject 的时刻为 e（每 cycle 最多接纳 ramp_bw 个到达）；
  3. 若 e > arrive，则 flit 在 [arrive, e) 内占用该节点的 eject 缓冲；
  4. ramp_buf = 所有节点、所有 flit 的这类等待区间中，同一时刻重叠 flit 数的最大值。

对比 link_buf：输出链路上因发射槽被占而等待（可部分转移到边界 AFIFO）；ramp_buf 发生在目的节点自身的下 ramp 前，无法用象限边界 AFIFO 代替。

有 ramp_buf 时，下 ramp 的「突发能力」是多少？

重要区分：本模型里下 ramp 的物理持续带宽恒为 ramp_bw flit/cycle —— 有缓冲也不会让单 cycle 吐出超过 ramp_bw 个 flit。 ramp_buf 描述的是可吸收的到达突发，不是 eject 加速。

令牌桶模型：在任意长度为 T 的窗口内，该节点下 ramp 最多可接收的到达 flit 数 ≤ ramp_bw · T + ramp_buf
等价地：相对持续速率 ramp_bw，最多可多攒 ramp_buf 个 flit 而不丢包（在日历意义下不破坏可行性）。
量含义16×16 示例
ramp_bw下 ramp 持续 eject 上限1 / 2 / 4 flit/cycle
ramp_buf = K单节点 eject 口最深排队 K 个 flitmultitree K≈121；border bi rb=1 时 K=74
瞬时 eject 速率每 cycle 仍 ≤ ramp_bw有 K 个排队时，需 ≥ ⌈K/ramp_bw⌉ cycle 排空
到达突发（可吸收）短窗口内可超过 ramp_bw 的超额到达最坏 ≈ ramp_buf flit（在同一 arrive 附近同时到达）
读 Pareto 表时：预算 K 约束的是 max(link_buf, ramp_buf) ≤ K。 例如 border bi、ramp_bw=1 时 ramp_buf=74 —— 表示某节点曾同时有 74 个 flit 等下 ramp， 持续 eject 仍为 1 flit/cy，但调度允许这 74 个 flit 在 eject 口排队； 要把该方案纳入 K≤2 的 Pareto，必须换 ramp_bw=2（此时 border 的 ramp_buf 降为 2） 或改用 hybrid B=2 等更平滑的到达模式。

本报告 Pareto 中的 pipe_mk 假设 eject 口确有 depth ≥ ramp_buf 的缓冲； 若硬件只给 K 个 slot，则 ramp_buf > K 的方案不可实现，即使其 pipe_mk 更小。

4. 关键拐点解读

  • K=0：仅 peak_buf=0：ramp_bw=1 → 单环 uni 1474；ramp_bw=2/4 → 单环 bi 754。
  • K≤2：ramp_bw=2 → border Q=4 bi 267；ramp_bw=1 → hybrid B=2 bi 418（border bi 需 ramp_buf=74）。
  • K≤2, rb=4：与 rb=2 相同拐点：border bi 267（更高 ramp_bw 吸收了 eject 突发，peak_buf 仍≤2）。
  • K≈10~30：hybrid B=16 / grid 4×2 开始领先 border：mk 251~201。
  • K≥56：grid 8×2 bi 最优 mk=189（rb=2/4）；仍高于 eject LB 128/64。
  • eject 地板：ramp_bw=1→255，2→128，4→64；仅当 fill/Lmax 已贴地板时 ramp_bw 才成为主导瓶颈。

7. 结论

  1. 在仅允许 router 缓冲 ≤K 时，不存在比 Pareto 前沿更小的 makespan； 更靠根的分叉（grid 8×2、multitree）确实更快，但必须付出 K 级缓冲。
  2. K≤2 的工程最优：ramp_bw=2/4 → border Q=4（267）； ramp_bw=1 → hybrid B=2（418）或 border uni（437）。
  3. K 充足时：grid 8×2 双向在 ramp_bw=2/4 下 mk=189；再往下受 Teject 限制（128/64），需提高 ramp_bw 而非继续加深分叉。
  4. eject 突发缓冲在目的节点下 ramp，无法转移到边界 AFIFO——这是 Q*≈H 的根本原因。
