Mesh 坐标系

Broadcast 广播

在 12×16 mesh 上，root 位于节点 0（左上角）。flit 从 root 的 up-ramp 注入后，沿_latency 最小生成树_的每条树边向子节点传播；router 在分叉处 fork 复制 flit。箭头方向即单 flit 在 mesh 链路（H=4cy / V=8cy）上的流动方向。最远节点 (11,15) 累计链路延迟 164cy，加 ramp 得 makespan=166。

healthy M=1: makespan=166, period=1

Reduce 归约

使用与 broadcast 相同的 latency 树，但箭头指向 root。参与归约的叶节点 PE 经 up-ramp 注入本地 flit，router 内 inline combine，沿 mesh 链路（H=4cy / V=8cy）逐级汇聚；最终结果经 root down-ramp 输出至 PE。M=1 makespan=166（1cy up + 164cy mesh + 1cy down，与 broadcast 对称）。

healthy M=1: makespan=166, period=1

Gather 收集

每个非 root 节点沿 Dijkstra 最短路径（H=4, V=8）向 root 发送 flit。 全局 link-time calendar：先为每个 flit 分配 root down-ramp 完成时刻 Fᵢ=t₀+i（相邻 slot 间隔 1 cycle），再反向逐 hop 在 (link, cycle) 网格上预约 send-time； 冲突则 Fᵢ 递增并重算后续 slot。下图为空间路径 + 时隙日历示意。

Lᵢ 的含义

Lᵢ 是将 191 个非 root 源按路径延迟升序编号后，第 i 个源的端到端延迟 （up-ramp 1cy + mesh 最短路径 + root down-ramp 1cy，与 PathLatency 一致）。 M=1 healthy 时 L₀=6（最近邻），L₁₉₀=166（角点 (11,15)）。

精确 makespan 下界：maxᵢ(Lᵢ−i)+(N−2)=205

root down-ramp 每 cycle 只能完成 1 个 flit，191 个 flit 占用连续 slot t₀, t₀+1, …, t₀+190。 第 i 个 flit 最早在 cycle Lᵢ 完成，故约束 t₀+i ≥ Lᵢ，即 t₀ ≥ Lᵢ−i。 取 t₀=maxᵢ(Lᵢ−i)=15，则 makespan = t₀+190 = t₀+(N−2) = 205。

为何 maxᵢ(Lᵢ−i)=15？

瓶颈不是最远角点 (11,15)（L=166 排在 i=190，166−190=−24，约束很松）， 而是「路径中等偏长、但排序靠前」的源：例如节点 6（坐标 (6,0)）在排序中 i=11， Lᵢ=26，故 Lᵢ−i=26−11=15。 含义：已有 11 个更短路径的源占用了 slot t₀…t₀+10，该源若要在 slot t₀+11 完成， 必须整体把 pipeline 起点后移 t₀=15 cycle，使 t₀+11≥26。 同 slack=15 的还有 i=15（节点 7）、i=19（节点 8）等 L=30/34 的 x 轴同行源。

闭式近似：(N−2)+max_path−mesh_diam+H+V=204

项 值 含义
N−2 190 191 个 root slot 的首尾索引差
max_path 166 最远源 up+mesh+down 全路径延迟
mesh_diam 164 纯 mesh 直径 11×H+15×V
max_path−mesh_diam 2 两端 ramp 各 1cy

代入 190+2+12=204，比精确式少 1 cycle（闭式用 max_path 代替全部 Lᵢ 分布）。 period=(N−1)×M=191 是 root 带宽下界（稳态重复间隔），不是 makespan。

healthy M=1: makespan=205, period=191

AllGather 全收集

最优方案不是 gather+broadcast。把数据汇经 root 再广播会串行化两阶段并制造 root 热点， 下界被高估为 205+166=371。真实瓶颈是每个节点的单条 down-ramp 必须吞入 (N−1)M flit， 故下界 = max((N−1)M, bisection, 最坏角节点 gather) = 205(M=1)， 约等于一次最坏 gather，而非 gather+broadcast。

实现的最优算法是 双向维序多树（bidirectional dimensional multi-tree，无字面意义的 Hamiltonian 环， 每维的「双向 line」即 mesh 上环的等价物）：每个源沿 X-先-Y 维序展开一棵组播树 —— 行脊（H-link 双向）+ 每列分叉（V-link 双向）。转发是 router 内 fork：节点把到达的 flit 复制一份下 down-ramp（eject 到 PE），另一份继续转发，中间节点从不 eject 后再 reinject， 因此不付 10cy 的 PE/SRAM bounce。N 棵树叠加后由全局 link-time calendar 贪心装填， 每条有向 link、每个 down-ramp 每 cycle≤1 flit，构造即无冲突。 最坏角节点 (0,0) 的列0 漏斗承载 180M flit、down-ramp 吞 191M flit，恰是瓶颈。 仿真（C++ 调度器 + utils/sim_dim_multitree.py）确认 makespan 精确命中下界： 205(M=1) / 769(M=4) / 3061(M=16) / 12229(M=64)，eff=1.0，优于 2D dimensional(235) 与 gather+broadcast(371)。 但这组 eff=1.0 数依赖充足缓冲（见下）。详见 report.html 的「AllGather 理论再分析」。

真实物理约束：零 eject 缓冲 + 有界网内等待

无冲突指同一 cycle 同一资源最多 1 flit（inject≤1 / eject≤1），不等于不排队。 硬件上有两条硬约束：(1) down-ramp/eject 缓冲必须为 0——到达当 cycle 即被 PE 取走； (2) 每跳网内等待必须有界 W——无界滞留会与后续 flit 踩踏（卡住的 flit 占住端口，后面流水 flit 撞上）。 之前 makespan=205/eff=1.0 依赖角节点 50–87 flit 的 eject 队列 + 端口无界滞留，正是被禁止的场景。

AllReduce 全归约

左：Reduce 阶段，红箭头由叶向 root 汇聚并在 router combine。右：Broadcast 阶段，蓝箭头由 root 向全网 fork。calendar 在 mesh 链路上为两阶段分别预分配时隙后拼接。

AllToAll 全交换

虚线为水平二分切面（x=5|6）。紫色双向箭头表示跨切面的 flit 交换（瓶颈链路）；淡紫水平线为行内 X 维交换。全网每对节点互发 M flit，calendar 分 X/Y 两阶段填满切面带宽，makespan 下界 768×M。

healthy M=1: makespan=929, period=764

AnyToAny 任意置换

每条彩色折线为一条置换路径 i→P(i)（Dijkstra 最短路径，沿 mesh 水平/竖直链路转弯）。模式事先已知，calendar 为每条路径的每一跳预分配 send-time，全网并行、链路无冲突。

healthy M=1: makespan=360, period=764

故障下离线重调度 — Mesh 上的数据流变化

故障后在同一 mesh 坐标系上更新拓扑（删节点/删边），Dijkstra 重算路径，calendar 全链路时隙表 offline 重生成。

节点故障：故障节点从 mesh 消失（✕）；若其为 root 则迁移至节点 1；broadcast 树在剩余 191 节点上重建，箭头拓扑整体改变。

链路故障：左图 0→1 走单跳 H-link（lat=4）；右图该边断开（红虚线✕），0→1 改走 0→12→1 两跳 V-link（橙箭头，lat=16），calendar 将对应路径上的 send-time 整体后移。
